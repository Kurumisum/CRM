# 电信CRM客户中心 - 统一语言术语表（Ubiquitous Language）

## 一、概述

统一语言（Ubiquitous Language）是领域驱动设计的核心概念之一，它确保技术团队和业务人员使用相同的术语来描述业务概念，避免沟通误解。

本术语表定义了电信CRM客户中心所有核心业务概念及其准确含义。

## 二、核心业务实体术语

### 2.1 客户相关（Customer Domain）

| 术语 | 英文 | 定义 | 备注 |
|------|------|------|------|
| 客户 | Customer | 与电信公司建立服务关系的主体，是合同签约方和责任承担方 | 客户是服务的购买者，但不一定是使用者 |
| 公众客户 | Individual Customer | 以个人身份开户的客户类型 | 也称"个人客户"，占客户总数80%以上 |
| 家庭客户 | Family Customer | 以家庭为单位开户的客户类型 | 支持家庭成员共享套餐、统一缴费 |
| 政企客户 | Enterprise Customer | 以企业/政府机关身份开户的客户类型 | 也称"集团客户"，通常有专属服务和优惠 |
| 客户档案 | Customer Profile | 客户的基本信息，包括姓名、证件、联系方式等 | 不同客户类型有不同的档案要求 |
| 客户编号 | Customer ID | 客户的唯一标识符 | 全局唯一，终身不变 |
| 客户等级 | Customer Level | 基于消费金额、在网时长等评定的客户级别 | 通常分为1-5星，等级越高享受的权益越多 |
| 客户状态 | Customer Status | 客户当前的服务状态 | 正常、欠费、停机、销户 |
| 开户 | Customer Registration | 客户与电信公司建立服务关系的过程 | 需要实名认证，一次开户可办理多个用户 |
| 销户 | Customer Termination | 终止客户与电信公司的服务关系 | 需要结清所有费用，注销所有用户 |

### 2.2 用户相关（User Domain）

| 术语 | 英文 | 定义 | 备注 |
|------|------|------|------|
| 用户 | User | 实际使用电信服务的载体，对应一个手机号码 | 用户必须归属于某个客户 |
| 个人用户 | Individual User | 单个号码使用的用户类型 | 占业务办理量90%以上 |
| 群用户 | Group User | 多个号码组成的用户群体 | 包括家庭群、政企集团 |
| 手机号码 | Phone Number | 11位数字的移动电话号码 | 全国唯一，是用户的主要标识 |
| 用户编号 | User ID | 用户的内部唯一标识符 | 区别于手机号码，用于系统内部 |
| 用户状态 | User Status | 用户当前的服务状态 | 预开、正常、停机、预销户、已销户 |
| 入网 | User Activation | 用户首次激活使用服务 | 也称"开户"或"开通" |
| 停机 | User Suspension | 暂停用户的服务 | 分为欠费停机和用户申请停机 |
| 复机 | User Resumption | 恢复已停机用户的服务 | 欠费停机需先缴费 |
| 销号 | User Termination | 注销用户号码 | 也称"销户"，号码进入冷冻期后回收 |
| 过户 | User Transfer | 将用户从一个客户转移到另一个客户 | 需要双方客户同意 |

### 2.3 号码资源相关

| 术语 | 英文 | 定义 | 备注 |
|------|------|------|------|
| 号码 | Number | 手机号码，11位数字 | 包括号段、归属地等属性 |
| 号段 | Number Segment | 号码的前缀段，标识运营商和业务类型 | 如130、131等 |
| 靓号 | Premium Number | 号码尾数有规律或特殊意义的号码 | 如AAAA、ABCD等，通常需要额外费用 |
| 归属地 | Home Location | 号码所属的地理区域 | 影响长途和漫游资费 |
| 携号转网 | Number Portability | 保留号码更换运营商 | 国家政策要求支持 |

### 2.4 SIM卡相关

| 术语 | 英文 | 定义 | 备注 |
|------|------|------|------|
| SIM卡 | SIM Card | 用户身份识别卡，存储用户信息和密钥 | Subscriber Identity Module |
| ICCID | Integrated Circuit Card ID | SIM卡的唯一标识号，20位数字 | 印刷在SIM卡背面 |
| IMSI | International Mobile Subscriber Identity | 国际移动用户识别码，15位数字 | 存储在SIM卡中，全球唯一 |
| IMEI | International Mobile Equipment Identity | 移动设备识别码，标识手机终端 | 15位数字，用于设备追踪 |
| 换卡 | SIM Card Replacement | 更换SIM卡，保留号码不变 | 常见原因：卡损坏、升级（如4G升5G） |
| 补卡 | SIM Card Reissue | SIM卡丢失后重新发卡 | 需要先挂失 |
| 挂失 | Report Loss | 声明SIM卡丢失，停止服务 | 可以解挂或补卡 |

### 2.5 账户相关（Account Domain）

| 术语 | 英文 | 定义 | 备注 |
|------|------|------|------|
| 账户 | Account | 用于费用管理的虚拟账户 | 一个客户可以有多个账户 |
| 账户编号 | Account ID | 账户的唯一标识符 | 全局唯一 |
| 预付费账户 | Prepaid Account | 先充值后使用的账户类型 | 余额不足时停机 |
| 后付费账户 | Postpaid Account | 先使用后付费的账户类型 | 通常有信用额度 |
| 账户余额 | Account Balance | 账户中的可用金额 | 实时更新 |
| 信用额度 | Credit Limit | 后付费账户允许透支的最大金额 | 根据客户信用评级确定 |
| 充值 | Recharge | 向账户存入金额 | 支持多种渠道：现金、银行卡、支付宝等 |
| 扣费 | Deduction | 从账户扣除金额 | 由计费系统触发 |
| 冻结 | Freeze | 暂时锁定一部分余额 | 用于预扣费、担保等场景 |
| 解冻 | Unfreeze | 释放冻结的余额 | 交易完成或取消时 |

### 2.6 资费相关

| 术语 | 英文 | 定义 | 备注 |
|------|------|------|------|
| 资费 | Tariff | 电信服务的定价标准 | 包括语音、流量、短信等 |
| 套餐 | Service Package | 打包销售的服务组合 | 通常包含月租费和一定量的免费使用 |
| 月租费 | Monthly Fee | 每月固定收取的费用 | 也称"月功能费" |
| 流量 | Data Traffic | 移动数据使用量 | 单位：MB、GB |
| 通话时长 | Voice Duration | 语音通话时长 | 单位：分钟 |
| 短信 | SMS | 短消息服务 | Short Message Service |
| 增值业务 | Value Added Service | 基础通信服务之外的附加服务 | 如彩铃、流量包等 |
| 套餐外资费 | Out-of-Package Rate | 超出套餐包含量后的计费标准 | 通常比套餐内贵 |

## 三、业务流程术语

### 3.1 开户流程

| 术语 | 英文 | 定义 | 备注 |
|------|------|------|------|
| 实名认证 | Real-name Authentication | 验证客户身份信息的真实性 | 国家强制要求 |
| 选号 | Number Selection | 从号码池中选择心仪的号码 | 靓号可能需要额外付费 |
| 选套餐 | Package Selection | 选择合适的资费套餐 | 可以后续变更 |
| 激活 | Activation | 首次使用，激活号码和服务 | 通常在开户后24小时内 |
| 预开 | Pre-activation | 已开户但未激活的状态 | 不产生月租费 |

### 3.2 变更流程

| 术语 | 英文 | 定义 | 备注 |
|------|------|------|------|
| 套餐变更 | Package Change | 更换资费套餐 | 分为立即生效和次月生效 |
| 生效规则 | Effective Rule | 业务变更何时生效的规则 | 立即生效/次月生效/指定日期生效 |
| 立即生效 | Immediate Effective | 办理后立即生效 | 可能产生补差费 |
| 次月生效 | Next Month Effective | 下月1日零时生效 | 最常见的生效方式 |
| 过户 | Ownership Transfer | 变更用户的归属客户 | 需要双方客户到场 |
| 资料变更 | Profile Update | 修改客户或用户的基本信息 | 某些信息变更需要实名认证 |

### 3.3 销户流程

| 术语 | 英文 | 定义 | 备注 |
|------|------|------|------|
| 预销户 | Pre-termination | 申请销户，进入等待期 | 通常等待期为30天 |
| 确认销户 | Termination Confirmation | 正式注销号码 | 不可恢复 |
| 冷冻期 | Freeze Period | 销户后号码的冷冻期 | 通常3-6个月，防止恶意注册 |
| 号码回收 | Number Recycling | 冷冻期后重新投入号码池 | 可以被新用户选择 |
| 欠费销户 | Arrears Termination | 因长期欠费而强制销户 | 需要先结清欠费才能再次开户 |

| 2 | 欠费停机 | Suspended-Arrears | 因欠费被停机 |

| 8 | 预销户 | Pre-termination | 已申请销户，等待期中 |

## 五、技术术语

### 5.1 架构设计

| 术语 | 英文 | 定义 | 备注 |
|------|------|------|------|
| 聚合 | Aggregate | DDD中的核心概念，一组相关对象的集合 | 有明确的边界和聚合根 |
| 聚合根 | Aggregate Root | 聚合的入口对象 | 外部只能通过聚合根访问聚合 |
| 实体 | Entity | 有唯一标识的领域对象 | 生命周期独立 |
| 值对象 | Value Object | 无唯一标识的领域对象 | 不可变，可替换 |
| 领域事件 | Domain Event | 领域中发生的重要事情 | 用于解耦和集成 |
| 限界上下文 | Bounded Context | 明确的概念边界 | 不同上下文中同一术语可能含义不同 |
| 防腐层 | Anti-Corruption Layer | 隔离外部复杂性的转换层 | 保护领域模型不受污染 |

### 5.2 集成术语

| 术语 | 英文 | 定义 | 备注 |
|------|------|------|------|
| 同步调用 | Synchronous Call | 实时等待响应的服务调用 | 用于强一致性场景 |
| 异步调用 | Asynchronous Call | 不等待响应的服务调用 | 用于解耦和提高性能 |
| 消息队列 | Message Queue | 异步通信的中间件 | 如Kafka、RabbitMQ |
| 事件总线 | Event Bus | 发布订阅模式的事件分发机制 | 用于领域事件传播 |
| API网关 | API Gateway | 统一的API入口 | 提供路由、认证、限流等功能 |
| 服务编排 | Service Orchestration | 中心化的服务调用协调 | 由编排器控制流程 |
| 服务协同 | Service Choreography | 去中心化的服务协作 | 通过事件驱动 |

### 5.3 性能相关

| 术语 | 英文 | 定义 | 备注 |
|------|------|------|------|
| 高并发 | High Concurrency | 系统同时处理大量请求的能力 | 本项目要求支持300W/天 |
| 限流 | Rate Limiting | 限制单位时间内的请求数量 | 保护系统不被压垮 |
| 熔断 | Circuit Breaker | 故障快速失败机制 | 防止故障扩散 |
| 降级 | Degradation | 关闭非核心功能以保证核心功能 | 应对流量洪峰 |
| 缓存 | Cache | 临时存储频繁访问的数据 | 提高响应速度 |
| 分库分表 | Sharding | 将数据分散到多个数据库/表 | 解决单库性能瓶颈 |
| 读写分离 | Read-Write Separation | 读操作和写操作使用不同的数据库 | 提高并发能力 |

### 5.4 数据一致性

| 术语 | 英文 | 定义 | 备注 |
|------|------|------|------|
| 强一致性 | Strong Consistency | 任何时刻读取的都是最新数据 | 性能较低，用于核心业务 |
| 最终一致性 | Eventual Consistency | 经过一定时间后数据会达到一致 | 性能高，用于非核心业务 |
| 补偿事务 | Compensating Transaction | 通过反向操作撤销已完成的事务 | 分布式事务的常用模式 |
| Saga模式 | Saga Pattern | 将长事务拆分为多个本地事务 | 每个事务有对应的补偿操作 |
| 两阶段提交 | Two-Phase Commit (2PC) | 分布式事务的经典协议 | 有性能和可用性问题 |
| TCC | Try-Confirm-Cancel | 分布式事务的一种实现方式 | 业务侵入性强但性能好 |

## 九、术语使用规范

### 9.1 命名规范
1. **统一使用术语表中的标准术语**，不使用口语化或地方方言
2. **技术文档优先使用英文术语**，业务文档优先使用中文术语
3. **代码中的命名必须使用英文术语**，保持与领域模型一致
4. **禁止使用缩写**，除非是业界通用的标准缩写（如ARPU、MOU）

### 9.2 上下文映射
同一术语在不同上下文中可能有不同含义，需要明确：

| 术语 | 客户上下文 | 用户上下文 | 账户上下文 |
|------|-----------|-----------|-----------|
| 开户 | 客户建立服务关系 | 用户激活号码 | 账户创建 |
| 销户 | 客户终止服务关系 | 用户注销号码 | 账户关闭 |
| 状态 | 客户服务状态 | 用户号码状态 | 账户使用状态 |

### 9.3 术语演进
随着业务发展，术语可能需要更新：
1. 新增术语需要团队评审
2. 修改术语定义需要影响分析
3. 废弃术语需要迁移方案
4. 保持术语表的版本管理

## 十、总结

本统一语言术语表涵盖了电信CRM客户中心的核心业务概念，为技术团队和业务团队提供了共同的语言基础。

**使用建议：**
1. 所有团队成员必须学习和使用统一语言
2. 在需求讨论、设计评审、代码实现中一致使用
3. 遇到术语歧义时，及时更新术语表
4. 定期组织术语培训，确保新成员掌握
5. 在代码评审中检查术语使用的正确性

**持续改进：**
- 本术语表是活文档，随业务发展持续更新
- 鼓励团队成员提出术语优化建议
- 每季度评审一次，确保术语的准确性和完整性
